# R2：参数、事件规则与状态版本冻结

> 文档类别：State Freeze Specification
> 所属阶段：R2
> 状态：草案 v0.2（event-zone 状态机修订版）
> 前置依赖：R1-T10 final gate、12 行 `R2_decision_matrix`、R0-T10 / R0-T15 正式状态产物与完整 lineage
> 后续阶段：R3 释放、风险集与标签设计
> 生效边界：R2 只冻结状态参数、双层状态机、事件区间规则、状态版本和 canonical 产物；不重新执行 R1 的结构发现与零模型，不使用未来收益、未来波动、未来方向、未来路径、release label、precision / recall、回测结果或交易结果选择参数。

v0.2 在 v0.1 的参数候选与冻结治理基础上，补全两层状态机：第一层为 R0 已定义的 daily confirmed-state 状态机，第二层为 R2 新增的 `d` qualification 与 `g` grouping event-zone 状态机。相应调整 R2-T03 至 R2-T08 的统计指标、决策输入、canonical schema、无前视回放和 R3 交接边界。

## 1. R2 阶段目标

R2 的目标是消费 R1 已完成的结构验证证据，将具备结构资格的候选状态收敛为可审计、可复现、可由 R3 唯一消费的冻结状态版本和事件接口。

R2 需要回答五个问题：

1. **参数候选如何收敛**：R1-T10 交接的 shared-q 与 layer-specific q-vector 候选中，哪些进入正式事件建模，哪些只保留为 strict core、fallback、sensitivity 或被拒绝。
2. **日度状态如何进入和退出**：在不改写 R0 `K=3` confirmation 语义的前提下，明确 confirmed state 的进入、维持、退出原因和可用时点。
3. **event zone 如何形成和关闭**：如何使用 `d` 将 atomic confirmed intervals 资格化，如何使用 `g` 将 qualified components 分组，如何处理短重入、链式合并、质量中断、开放区间和删失。
4. **哪些状态版本最终冻结**：对四条平等候选路线执行 hard gate、Pareto / complexity 比较和人工决策，允许冻结 0–4 个正式版本。
5. **如何形成 canonical 产物**：只物化最终选中的版本，完成双层状态机回放、异常扫描、独立审阅和最终 freeze manifest，再向 R3 提供唯一正式入口。

R2 的结论只能表述为状态和事件接口的冻结结论，例如：

```text
某个 state_line × W × q-vector × K × d × g 已通过 R2 门禁，
其 daily confirmed-state 与 event-zone 状态机已被冻结为 R3 合法输入。
```

R2 不得表述为：

```text
该状态未来会上涨；
该窗口对应确定的交易周期；
confirmed state 退出等同于 release；
该版本具有交易优势；
该参数是数学或市场意义上的全局最优；
某个 future direction / return label 已经成立。
```

## 3. 已确认的 R2 研究取舍

### 3.1 四条 q-vector 路线平等进入事件建模

R2-T01 的 primary shortlist 固定为四条路线：

```text
S_PCT  × W120 × K3 × qP=.20,qC=.20,qT=.25,qV=.20
S_PCT  × W250 × K3 × qP=.20,qC=.20,qT=.25,qV=.20
S_PCVT × W120 × K3 × qP=.20,qC=.20,qT=.20,qV=.30
S_PCVT × W250 × K3 × qP=.20,qC=.20,qT=.20,qV=.30
```

四条路线以平等身份进入 R2-T03 的 `d×g` 扫描和 R2-T04 的最终处置。R2 不预先指定：

```text
S_PCT 必须选择 W120；
S_PCVT 必须选择 W250；
W120 或 W250 是另一窗口的优胜者或等价表达。
```

现有 R1 证据只证明 W120 与 W250 均有结构资格，并且状态身份存在明显差异；没有证明某个窗口天然属于某条状态线。

### 3.2 q-vector 是 primary candidate，shared-q 是 strict core / fallback reference

R1-T14 的实际结果显示，qT=.25 与 qV=.30 带来 material 的状态覆盖和区间几何变化，并通过相应正式结构门禁。因此，四个 q-vector center 作为 R2 的 primary freeze candidates。

对应 shared-q：

```text
qP=qC=qT=qV=.20
```

在 R2 中承担两种角色：

1. **strict core reference**：当对应 q-vector 被选中时，shared-q 只作为同一 `state_line × W` 内部的严格核心标记，不生成额外独立产品版本。
2. **fallback candidate**：如果对应 q-vector 被拒绝，且 shared-q 的完整 event-zone sidecar 通过全部 hard gate，T04 可以将 shared-q 选为正式版本。

因此，T03 除四条 primary route 的 36 个 `d×g` cells 外，还必须对四个 shared-q reference 执行完整 36-cell fallback sidecar。总计：

```text
36 primary q-vector cells + 36 shared-q fallback cells = 72 cells。
```

约束为：

```text
retain_as_strict_core_only
→ 对应 state_line × W 的 q-vector 必须 selected。
```

同一 `state_line × W` 下，默认不得同时将 q-vector 和 shared-q 冻结为两个独立产品版本。若确有必要，必须在用户决策记录中提供明确 override justification，并通过独立科学审阅。

### 3.3 qT=.30 与 qV=.25 的边界

```text
qT=.30：保留为 immediate-neighbor sensitivity evidence，默认不进入 primary T03 shortlist。
qV=.25：R1-T10 已标记 do_not_freeze，R2 不得复活为冻结候选。
```

qT=.30 可以支持 qT=.25 的邻域连续性判断，但不能因 coverage 更高或 fragment 更低自动替代 qT=.25。若 R2 希望将其升级为 primary candidate，必须单独修改 R2-T01 shortlist contract，并重新评估完整 T03/T04 family；不得在看到 T03 结果后临时加入。

### 3.4 W120 与 W250 是独立窗口路线

W120 与 W250 使用不同的历史参考尺度，R2 将其治理为：

```text
W120 track：short-reference state track
W250 track：long-reference state track
```

这些名称只描述历史参考窗口，不预先宣称交易周期、持有周期或 release 速度。

同一状态线的两个窗口：

```text
不要求存在集合包含关系；
不能因 Jaccard 低而淘汰；
不能在冻结包中描述为同一状态的等价实现；
若同时冻结，必须使用不同 state_version_id；
必须向 R3 移交 overlap / intersection 信息。
```

### 3.5 不产品化任何额外 PCT parent 状态

R2 不建立、不冻结、不物化：

```text
S_PCT_parent_depletion_v1
```

或任何为了支撑 `S_PCVT` 而增加的 PCT parent 产品版本。

R1 中的 same-parameter PCT parent 只用于 `V_given_PCT`、nested null 和 parent-child 审计。R2 可引用相应 R1 evidence，但不将其升级为状态产品、event table 或 R3 入口。

`strict_core_member` 必须是**同一状态线内部**的严格阈值标记：

```text
S_PCT qT=.25 的 strict core = 同 W/K 的 S_PCT qT=.20；
S_PCVT qV=.30 的 strict core = 同 W/K 的 S_PCVT qV=.20。
```

它不是 S_PCT / S_PCVT 之间的 parent 产品关系。

### 3.6 `K/d/g` 的固定语义

所有 primary q-vector 与 shared-q reference 均使用 `K=3`。R2 不重新扫描 K。

```text
K：daily state confirmation。
   连续 3 个 raw true 日后，第 3 日成为第一个 confirmed_state=true 日；不得回填前两日。

d：atomic confirmed interval qualification threshold。
   atomic interval 中 confirmed_state=true 的交易日数必须 ≥ d，component 才取得事件资格。

g：event-zone grouping tolerance。
   qualified components 中间最多允许 g 个 raw_false_gap_day，
   才可归入同一个 retrospective event zone。本次修订属于同一
   R2-T02 protocol freeze 的研究者协议决策修订，不是新的研究阶段。
```

正式网格为：

```text
d ∈ {1,2,3}
g ∈ {0,1,2}
```

`d` 必须使用 `≥d`，并且只按 confirmed true trading days 计数。下列日不得计入 d：

```text
raw true but not confirmed；
unknown；
blocked；
diagnostic_required；
bridged gap；
calendar non-trading day。
```

`g` 的冻结计数对象为：

```text
raw_false_gap_day
= eligible=true
  AND quality_state=valid
  AND raw_state=false
```

同一个 pending event zone 内：

```text
g_used
= 从前一 qualified component 退出观察开始，
   到下一 component 达到 d 资格之前，
   累计观察到的 raw_false_gap_day 数量。
```

`g` 统计累计 raw=false 日，不是最大连续 raw=false 日。`raw_state=true AND confirmed_state=false` 属于 K confirmation 前置阶段，不增加 `g_used`，也不清零已经累计的 `g_used`。退出日的 `raw_state=false` 行计入 `g`；第 `g+1` 个 raw=false 行一旦可见，当前 zone 立即不可逆 finalized，`zone_finalization_time` 等于该行实际 `available_time`。calendar non-trading day 不计入 `g`。

以下仍为 hard break，不得计入 `g`，也不得 bridge：

```text
raw_state unknown/null
quality_state=unknown
quality_state=blocked
quality_state=diagnostic_required
eligible=false
missing observation
missing expected trading row
```

### 3.7 第一层：daily confirmed-state 状态机

R2 冻结并复用 R0 的 confirmation 语义：

```text
RAW_NOT_CONFIRMED
    ↓ 连续 K=3 个 raw true
CONFIRMED_ACTIVE
    ↓ 首个 raw non-true
CONFIRMED_EXITED
```

规则为：

```text
进入：第 K 个连续 raw true 日首次 confirmed_state=true，不回填前 K-1 日。
维持：raw_state=true 时 confirmed_state 持续为 true。
退出：raw_state=false / unknown / diagnostic_required / blocked 时，当日 confirmed_state=false。
confirmed_end_date：退出前最后一个 confirmed_state=true 日。
exit_observation_time：首次观察到 confirmed_state 已退出的交易日。
```

退出原因必须保留：

```text
natural_state_exit          # raw_state_false
quality_interruption        # unknown / diagnostic_required / blocked / missing observation
sample_end_censoring        # end_of_input_open
```

`confirmed_state` 退出只表示收敛状态不再被确认，不等同于 R3 的 release。

### 3.8 第二层：event-zone 状态机

R2 在 atomic confirmed intervals 之上冻结以下状态机：

```text
COMPONENT_FORMING
    ├─ confirmed_day_count 达到 d → QUALIFIED_ACTIVE
    └─ 未达到 d 即结束 → UNQUALIFIED_CLOSED

QUALIFIED_ACTIVE
    ↓ atomic confirmed interval 结束
GAP_PENDING
    ├─ 第 g+1 个 raw_false_gap_day 出现 → FINALIZED
    ├─ unknown / blocked / diagnostic / missing → FINALIZED_WITH_QUALITY_BREAK
    ├─ raw_state=true 且 confirmed_state=false → 进入或维持 preconfirmation，g_used 不变
    ├─ 新 confirmed component 开始 → REENTRY_PENDING_QUALIFICATION
    └─ 样本结束 → RIGHT_CENSORED

REENTRY_PENDING_QUALIFICATION
    ├─ 新 component 达到 d → 合并至同一 event_id，回到 QUALIFIED_ACTIVE
    ├─ 未达到 d 即结束 → 记录 unqualified reentry，并关闭前一 zone
    ├─ quality break → FINALIZED_WITH_QUALITY_BREAK
    └─ 样本结束 → RIGHT_CENSORED
```

执行顺序固定为：

```text
atomic confirmed interval
→ d qualification
→ qualified component
→ g grouping
→ event zone
```

第 `g+1` 个 raw=false 的 finalization 必须发生在该 raw=false 行的实际 `available_time`，不得等待后续 confirmed component，不得在后续 raw=true 出现后恢复，且 finalization 不可逆。

event-zone gap rows 必须分三类冻结：

```text
raw-false bridge row:
  eligible=true
  quality_state=valid
  raw_state=false
  confirmed_state=false
  增加 g_used；成功 merge 后 event_zone_member=true,
  is_raw_false_bridge=true, is_preconfirmation_gap=false,
  is_bridged_gap=true。

preconfirmation raw-true row:
  eligible=true
  quality_state=valid
  raw_state=true
  confirmed_state=false
  不增加 g_used，不重置 g_used；成功 merge 后
  event_zone_member=true, is_raw_false_bridge=false,
  is_preconfirmation_gap=true, is_bridged_gap=false。

confirmed component row:
  confirmed_state=true
  按 d qualification 处理，不属于 gap count。
```

`is_bridged_gap` 的定义冻结为：

```text
is_bridged_gap == is_raw_false_bridge
```

不得将 preconfirmation raw-true row 标为 `is_bridged_gap=true`。

未达到 d 的 confirmed interval 必须单独记录为：

```text
unqualified_confirmed_interval
```

它不是 false gap，也不能被 `g` 静默吞并。

### 3.9 `g` 的传递性、激活次数与 anti-percolation

对同一 `state_version_id × security_id`，qualified components 按 `gap≤g` 做传递性合并，形成 maximal connected event zone：

```text
A --gap≤g-- B --gap≤g-- C
→ A/B/C 属于同一 event zone。
```

默认不限制 bridge activation 次数，不预设 `max_bridge_count`。但必须记录：

```text
component_interval_count
bridge_count = component_interval_count - 1
bridged_gap_days
max_single_gap_days
zone_trading_span
zone_confirmed_day_count
confirmed_density = zone_confirmed_day_count / zone_trading_span
bridged_gap_ratio = bridged_gap_days / zone_trading_span
```

T03 必须执行 anti-percolation anomaly scan。若 unlimited chaining 导致 material geometry degeneration，才可通过独立 task 触发：

```text
max_bridge_count ∈ {1,2,unlimited}
```

的结构 sidecar。不得在无实际退化证据时新增该参数，也不得使用未来结果选择 bridge budget。

### 3.10 qualification、finalization 与 as-of 可用时点

同一事件至少区分：

```text
confirmed_start_date
first_qualification_time
last_confirmed_end_date
last_exit_observation_time
zone_finalization_time
membership_available_time
```

若 `d>1`，一个 atomic interval 的前 `d-1` 个 confirmed days 在回溯上属于最终 qualified component，但在当时尚未取得事件资格。因此必须同时保留：

```text
retrospective_component_member
component_qualified_as_of
```

若 `g>0`，完整 event-zone membership 可能在后续 component 出现后才最终确定。R2 不得把 `zone_finalization_time` 或 `membership_available_time` 回填到较早日期。

### 3.11 左删失、右删失与开放状态

必须独立标记：

```text
left_censored_component
right_censored_component
prequalification_right_censored
open_gap_pending_zone
right_censored_zone
```

规则为：

```text
左删失 component 不得被当作正常 onset；
右删失 component / zone 不得被当作自然退出；
prequalification_right_censored 不计入 short_interval_drop_rate denominator；
样本末尾仍处于 gap pending 时不得假装 zone 已 finalized。
```

年份边界不自动切断 event zone。周末和节假日不计入 g；g 按该证券的 eligible observed trading days 计数。

### 3.12 event ID 稳定性与 zone revision

`event_id` 必须在首次 qualified component 形成时确定，并在后续合并中保持稳定：

```text
event_id
= state_version_id + security_id + first_qualified_component_identity
```

后续 component 加入只更新：

```text
zone_revision
component_interval_count
latest_component_end
zone_status
membership_available_time
```

不得因后续合并而重排历史 event ID，也不得跨 `state_version_id` 合并。

### 3.13 `confirmed_state`、事件资格与 R3 risk set

R2→R3 contract 必须区分：

```text
state_risk_set_eligible
= row 在 evaluation time 已可见
  AND eligible=true
  AND quality_state=valid
  AND confirmed_state=true

qualified_event_risk_set_eligible
= state_risk_set_eligible=true
  AND event_zone_member=true
  AND component_qualified_as_of=true
  AND is_raw_false_bridge=false
  AND is_preconfirmation_gap=false
```

因此：

```text
R3 研究 confirmed 状态暴露时可使用 state_risk_set_eligible；
R3 研究 qualified event 发生后时必须使用 qualified_event_risk_set_eligible；
event_zone_member 不得替代上述任一风险集。
```

机器断言至少包括：

```text
state_risk_set_eligible → confirmed_state=true；
event_zone_member=false 但 confirmed_state=true 仍可 state_risk_set_eligible=true；
unqualified_reentry_member=true 且 confirmed_state=true → state_risk_set_eligible=true
  且 qualified_event_risk_set_eligible=false；
qualified_event_risk_set_eligible → state_risk_set_eligible=true；
is_raw_false_bridge=true → confirmed_state=false；
is_preconfirmation_gap=true → qualified_event_risk_set_eligible=false；
is_bridged_gap == is_raw_false_bridge；
event_zone_member=true 不推出任一 risk_set_eligible=true。
```

### 3.14 冻结数量不预设

R2-T04 可能最终选择：

```text
4 个版本；
3 个版本；
2 个版本；
1 个版本；
0 个版本。
```

零冻结版本是合法 R2 结论。此时 R2 可以完成验收，但 R3 保持关闭。

## 4. R2 决策层级

R2 不是新的结构显著性搜索阶段。R2 的正式决策采用四层结构。

### 4.1 Hard gate report

hard gate 由程序自动生成，不允许人工 override。检查对象为：

```text
candidate × d × g × denominator_scope
```

至少检查：

```text
R1 handoff status 合法；
input lineage / manifest / schema / hash 通过；

### 4.2 Pareto + complexity evaluation

通过 hard gate 的 cells 进入自动 comparison table。禁止使用未预注册的加权总分。

**Level 1：三层状态机与事件几何 Pareto 比较**

strict-core / expansion-shell 构成；
年份集中度；
R1 affected Delta / Lift；
global / nested null；

state identity / overlap。
```

以下指标只作门槛或分布判断，不按单向“越大越好”优化：

```text

zone duration；
zone span；
component count。
```

其 admissible range 或判定方向必须在 T02 预注册。

**Level 2：复杂度和治理成本**

```text

→ fewer unresolved warnings

```

简洁性只作为后置 tie-break，不能否定已经超过预注册 robust envelope 的 material 改善。

## 6. 候选与版本处置定义

### 6.1 `selected`

候选至少有一个 `d×g` cell 通过全部 hard gate，并经自动推荐、用户决策和独立审阅后被纳入 freeze plan。T05 必须物化，T06 必须验收，最终是否 frozen 取决于实际 artifact 是否通过 T06。

### 6.2 `rejected`

候选因 hard gate fail、所有 `d×g` cells 失败、Pareto / complexity 劣势、不可接受 warning 或用户明确拒绝而不进入 T05。必须保留 primary reason、evidence 和替代候选，不得从 registry 删除历史记录。

### 6.3 `retain_as_strict_core_only`

shared-q 不作为独立产品版本，只作为被选 q-vector 同状态线、同 W/K 下的严格核心标记。

必须满足：

```text
对应 q-vector disposition = selected；
shared-q 是 q-vector 的严格子集；
subset / count / identity 守恒通过；
T05 只写 strict_core_member 字段，不生成额外 state_version_id 或 event_id。
```

### 6.4 `needs_revision / blocked`

它不是最终候选 disposition，而是任务或 freeze plan 的失败状态。任何 unresolved anomaly、lineage failure、hard gate contract 缺失、状态转移不闭合、as-of leakage 或用户决策不完整都应阻止下游推进。

## 7. R2 验收条件

R2 的最低验收门槛如下：

1. **输入门槛**：R1-T10 final gate、12 行矩阵、R0-T10 / R0-T15 lineage、schema、hash 全部闭合。
2. **shortlist 门槛**：四条 primary q-vector route、shared-q strict-core / fallback、qT=.30 sensitivity 和 qV=.25 exclusion 均有确定角色。
3. **协议门槛**：daily confirmed-state、d qualification、g grouping、退出、短重入、删失、event ID、as-of time、指标和 hard gates 在 T03 前冻结。
4. **事件状态机门槛**：四条 primary 和四条 fallback reference 完成 72-cell scan；三层指标、状态转移、窗口 overlap、actual result analysis 和 anomaly scan 完整。
5. **决策门槛**：自动 hard gate、Pareto / complexity、用户决策和 freeze plan 完整；hard gate 不能被人工覆盖。
6. **物化门槛**：T05 只物化 selected 版本，数量与 T04 decision 完全一致，并忠实实现双层状态机。
7. **无前视门槛**：confirmation、qualification、reentry、finalization、revision、membership availability 和两类 risk-set 字段全部通过逐日回放。
8. **退出与删失门槛**：natural exit、quality interruption、left/right censor、open-zone 状态和 R3 可用范围均明确。
9. **冻结门槛**：最终 freeze manifest 只能引用 T06 已验收的实际 artifacts。
10. **审阅门槛**：formal task 的实际结果分析、anomaly scan、独立科学审阅和 repository final gate 完成。
11. **零版本边界**：允许零版本冻结；此时 R2 可关闭，但不得放行 R3。

## 10. 必须 task 与可选扩展的边界

R2-T01 至 R2-T08 是 R2 的最低必要路线。当前不单列额外可选研究 task，避免在冻结阶段继续扩展候选空间。

以下情形必须新建独立 task 或退回上游，而不能在既有 R2 task 中临时扩大范围：

```text
希望将 qT=.30 升级为 primary candidate；
希望复活 qV=.25；
希望新增 W500 或 K2/K5；
希望修改 q-vector；
希望改变 P/C/T/V 指标、阈值或 AND 逻辑；
希望新增未来 release / return / precision / recall 评价；
希望在未观察到 percolation 前预设 max_bridge_count；
希望改变先 d qualification、后 g grouping 的顺序。
```

回退原则：

```text
指标、W/q/K、状态逻辑变化 → 最低退回 R0，并重新进入 R1；
R1 零模型、稳定性或样本设计变化 → 退回 R1；
d/g、confirmed exit、event-zone transition、删失、event ID 或 as-of 规则变化 → 退回 R2-T02；
T04 selected list 或 selected d/g 变化 → 退回 R2-T04，并重跑 T05–T08；
release、标签、matched control 或 outcome risk-set exclusion 变化 → R3。
```

## 11. R2 到 R3 的交接原则

R2 交给 R3 的不是“交易信号”，而是具有唯一版本、明确可用时点和严格风险集边界的状态与事件接口。

R3 必须遵守：

```text
只读取 R2 final freeze manifest 登记的版本；
只用 state_risk_set_eligible 构造 confirmed-state 暴露；
只用 qualified_event_risk_set_eligible 构造已取得事件资格后的暴露；
不得以 event_zone_member、retrospective component membership 或 bridged gap 扩张风险集；
不得将 quality_interruption 或 right-censor 当作自然 release；
不得用 zone_finalization_time 回填更早退出日；
W120 / W250 是独立窗口路线，不预设交易周期；
若两个窗口同时冻结，必须显式处理同证券同日期和 event-zone 重叠；
strict_core_member 只能作为同一冻结版本内部的严格度分层；
selection_path_not_independently_confirmed 必须进入 R3 limitation；
不得读取 rejected、sensitivity-only 或未物化候选。
```

R3 必须明确选择 release study 的时间口径：

```text
provisional component-exit：从 last_exit_observation_time 起算，
允许实时响应，但必须接受后续 g 容差内重新进入同一 zone 的可能；

finalized-zone：从 zone_finalization_time 起算，
该口径实时可执行且已确认 zone 关闭，但信号天然延迟；

回溯 final-exit：可把 last_confirmed_end_date 或 last_exit_observation_time
作为历史事件日期，但必须显式处理后续 g 日无重入带来的事后确认和前视风险。
```

R3 才能定义和冻结：

```text
release；
future label；
risk-set exclusion；
matched control；
outcome horizon；
precision / recall；
与 outcome window 有关的 cooldown。
```

R2 的最终含义是：

> R1 证明候选状态具备结构资格；R2 决定哪些参数、daily confirmed-state 规则和 event-zone 规则可以被冻结，并把被选版本物化成可按时点消费的 canonical 状态与事件接口；R3 再研究这些已定义的状态退出何时构成 release，以及退出后发生什么。
