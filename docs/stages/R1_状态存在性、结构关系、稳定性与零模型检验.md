# R1 状态存在性、结构关系、稳定性与零模型检验

> 文档类别：R1 Structural Validation Specification
> 所属阶段：R1
> 协议版本：R1 v0.2 / R1-T01 manifest lock
> 前置依赖：R0-T11 evidence-bound handoff package
> 后续阶段：R2 状态冻结讨论
> 生效边界：R1 只做结构验证，不定义未来标签，不使用未来收益、未来波动扩张、未来突破方向、回测收益或交易结果选择指标及参数；R1 不冻结最终状态版本，R2 才能冻结最终 `W/q/K`、状态线版本、确认规则、区间规则和后续事件接口。

## 1. R1 阶段目标

R1 是 Structural Validation Gate，目标是验证 R0 产出的候选状态是否存在、是否有结构、是否稳定、是否超过零模型解释。R1 的合法结论是结构性结论，例如某条状态线与配置具备进入 R2 冻结讨论的证据，或因输入前提不成立必须退回 R0；R1 不能宣称交易优势、释放方向、未来收益、组合表现或最终冻结状态。

R1 必须分线研究 `S_PCT` 与 `S_PCVT`。`S_PCT = P ∧ C ∧ T`，表示不要求参与度枯竭的结构收敛状态；`S_PCVT = P ∧ C ∧ T ∧ V`，表示在 `S_PCT` 基础上叠加 V 枯竭约束的结构收敛状态。两条状态线可以在后续 evidence 中形成 state-line-specific candidate，例如 `S_PCT_fast_candidate` 或 `S_PCVT_depletion_candidate`，但这些候选只有在 R1-T03 至 R1-T09 产生正式证据后才能进入 R2 decision matrix。

## 2. 阶段定位与依赖边界

R1 只能消费 R0-T11 交接中允许 R1 使用的 evidence-bound package，包括 R0 候选状态日表、确认区间表、manifest、config hash、schema、quality flags 与 `confirmation_time`。R1-T01 不读取 raw / external / MarketDB / `.day`，不绕过 R0 授权交接产物，不重新计算 strict-past percentile，也不生成 future label、future return、release direction、breakout direction、backtest、portfolio 或 trade signal。

R1 对 raw 与 confirmed 采用 raw/confirmed 双主线。`raw` 用于诊断定义本体、单指标边界、层内互补性和 raw fragment；`confirmed` 用于判断是否具备 R2 `freeze_candidate` 或 `review_candidate` 资格。R2 decision matrix 的主状态必须以 `confirmed_state` 与 `confirmation_time` 为准，即 `r2_decision_basis = confirmed_state`。raw 结果只能作为诊断、解释或 warning 来源，不能单独使某配置进入 `freeze_candidate`。

R0-T11 已记录当前 R0 formal package 的 confirmed interval 为 0，这是 R1 的 input fact 和后续分析限制。R1 不得在本阶段修改 K、回填 confirmed state、做 gap merge、降低阈值或重跑 R0 来制造 confirmed intervals；若后续发现 R0 manifest、schema、hash、confirmation_time、严格过去分位、unknown/blocked 语义或候选状态表/确认区间表对齐不成立，应输出 `blocked_return_to_R0`。

## 3. 已确认的 R1 研究取舍

R1 锁定 R0 主网格配置：`W = [120, 250, 500]`、`q = [0.10, 0.20, 0.30]`、`K = [2, 3, 5]`，并继续尊重 R0 manifest 中的 `dimension_rule = weak` 与 `weak_delta = 0.10`。这 27 组配置不是按未来表现选择参数的搜索池，不能跑完后按单组最小 p-value 或最大 JointLift 直接选参数。

`W=250,q=0.20,K=3` 的配置编号为 `W250_q20_K3`，在 R1 中只是 reference baseline，不是默认最优参数。27 组配置承担三类作用：全量轻量结构扫描、预注册配置正式验证、触发式全量零模型。正式零模型优先覆盖 `S_PCT @ W=250,q=0.20,K=3`、`S_PCT @ W=120,q=0.20,K=3`、`S_PCVT @ W=250,q=0.20,K=3` 与 `S_PCVT @ W=120,q=0.20,K=3`；触发式 sidecar 注册 `S_PCT @ W=120,q=0.20,K=2`、`S_PCVT @ W=500,q=0.20,K=3` 与 `S_PCVT @ W=250,q=0.20,K=5`。只有 baseline / challenger 结论冲突，或 R2 明确要求 family-level 选择证据时，后续 R1-T10 才允许对 27 组补跑全量零模型；即便补跑，也必须采用 family-level correction 或 max-statistic null。

基础稳定性只按年份检查，年份由交易日派生：`year = YEAR(trading_date)`。市场状态、流动性层、股票年龄、行业、交易约束和质量归因不进入 R1 v0.2 的基础门槛，除非后续数据层提供版本化、时点一致、可审计字段；没有 as-of 字段时，不得硬造这些分层，也不得把它们写成阻断性验收条件。

## 4. R1 primary hypothesis test

R1 的正式零模型预注册为 `P_fixed_independent_CTV_circular_shift`。其口径是 P 保持不变；C、T、V 在 `security_id × year × continuous trading segment` 内分别 circular shift；保留每层边际分布、块内自相关和持续结构；破坏 P/C/T/V 的 contemporaneous alignment。

正式置换次数为 `N_perm = 2000`，经验 p 值公式为 `p = (n_extreme + 1) / (N_perm + 1)`。主证据是 JointLift 与有限样本 empirical p，z-score 只能作为描述统计，不作为主结论。固定滞后集合为 `lag_set = [1, 3, 5, 10, 20]`，后续滞后结构关系不得临时扩展为参数搜索。

可选 sidecar 包括 `CTV-bundle shift`、无锚平移和块长 B 变体，但它们不替代主口径。sidecar 只在协议中登记触发条件，不在 R1-T01 执行。

## 5. R1 工作路线

R1 的工作路线遵循“先接收与复检、再轻量结构扫描、再分线画像与诊断、再正式零模型、再年份稳定性、最后交接矩阵”的顺序。R1-T01 合并后最多只能进入 R1-T02，不能直接进入 R1-T03、R1-T07 或 R2。

R1-T02 负责 R0 产物接收、lineage 与无前视复检，确认 R1 只能消费哪个 R0 evidence-bound package，并复核 manifest、schema、config hash、input hash、strict-past percentile、unknown/blocked 语义与 `confirmation_time`。R1-T03 至 R1-T06 才开始轻量结构扫描、状态画像、单指标诊断、层间条件 Lift 与固定滞后结构关系。R1-T07 执行预注册配置同步性零模型，R1-T08 检查年份稳定性与状态集中度，R1-T09 汇总 R1 验收门禁与 R2 交接矩阵。

## 6. R2 交接状态定义

每个 `state_line × candidate_config_id` 未来必须被分为四类之一：`freeze_candidate`、`review_candidate`、`do_not_freeze` 或 `blocked_return_to_R0`。R1-T01 只定义状态语法，不给 27 组配置打最终结论。

`freeze_candidate` 表示该状态线与配置通过 R1 结构门槛，具备进入 R2 冻结讨论资格，但不是 frozen。`review_candidate` 表示有一定结构证据，但存在 material warning，需要 R2 或补充 sidecar 审查。`do_not_freeze` 表示当前 R0 产物与 R1 协议下不具备进入 R2 冻结讨论资格。`blocked_return_to_R0` 表示 R1 无法合法解释该配置结果，原因是 R0 manifest、schema、hash、confirmation_time、严格过去分位、unknown/blocked 语义或候选状态表/确认区间表对齐等输入前提不成立，必须退回 R0 修复。

## 7. R1 验收条件

R1 验收必须同时满足协议、代码、数据 lineage 和结论纪律。所有正式结果必须能追溯至 R0 evidence-bound handoff、数据版本、代码提交、配置哈希、运行环境、输入哈希和输出哈希；任何未通过门禁的结果只能标记为探索性，不得作为正式证据。

R1 不允许使用未来收益、未来波动扩张、未来突破方向、未来路径或回测收益选择指标、参数或事件规则。R1 不允许把 raw 单独作为 R2 freeze basis，不允许把 baseline 写成最优参数，不允许将 confirmed interval 0 行解释成 R0 失败，也不允许在 R1 内修补 R0 输入前提后继续发布结果。

## 8. 输出 Schema 与 artifacts

R1-T01 输出 `R1_config_manifest`、对应 JSON Schema、validator、thin wrapper、契约测试、task 文档、stage 文档、evidence 与 README gate 更新。后续 R1 task 预计生成 light profile summary、state-line profile、single-layer diagnostic、conditional Lift、lag structure、null model result、year stability summary 与 R2 decision matrix，但这些产物不在 R1-T01 生成。

R1 evidence 只能记录路径、hash、counts、validator result 和 gate，不得嵌入 row payload，不得复制 DuckDB、Parquet、CSV 或 JSONL 内容。没有 manifest 的产物不得进入 validated、frozen 或 released。

## 9. PR 级 task 路线图

必须 task 路线图如下：

```text
R1-T01 验证协议、状态线假设与 manifest 锁定
R1-T02 R0 产物接收、lineage 与无前视复检
R1-T03 27 组 W/q/K 全量轻量结构扫描
R1-T04 S_PCT 与 S_PCVT 分线状态画像
R1-T05 单指标诊断与层内互补性分析
R1-T06 层间条件 Lift 与固定滞后结构关系
R1-T07 S_PCT/S_PCVT 预注册配置的同步性零模型
R1-T08 年份稳定性与状态集中度检查
R1-T09 R1 验收门禁与 R2 交接矩阵
```

可选 / 触发式 sidecar 如下：

```text
R1-T10 27 组全量零模型 family-level sidecar
R1-T11 CTV-bundle、无锚平移与块长 B 对照零模型
R1-T12 替代指标口径 sensitivity sidecar
```

## 10. 必须 task 与可选 task 的边界

R1-T01 至 R1-T09 是进入 R2 冻结讨论前的必须 task。R1-T10 至 R1-T12 是可选或触发式 sidecar，只有 baseline / challenger 冲突、R2 明确要求 family-level 选择证据、主零模型结论对 sidecar 假设敏感，或替代指标口径对状态语义构成 material warning 时才启动。

可选 sidecar 不能替代主验证路线，也不能作为绕过 R1-T02 至 R1-T09 的捷径。sidecar 产物若影响 R2 decision matrix，必须以独立 evidence、manifest、schema 和 validator 记录。

## 11. R1 到 R2 的交接原则

R1 到 R2 的交接只能提交结构验证证据和 `state_line × candidate_config_id` 的 decision status。R1 不冻结最终状态版本，不定义未来标签，不启动释放研究，不启动回测或交易约束验证。R2 只能在 R1-T09 完成后，根据 confirmed state、`confirmation_time`、R1 evidence、warning 与 sidecar 触发结果讨论 freeze candidate；PR 合并 R1-T01 后不得进入 R2。
