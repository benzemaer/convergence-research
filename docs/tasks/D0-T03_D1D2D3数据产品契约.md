# D0-T03：D1 / D2 / D3 数据产品契约

> Task ID：D0-T03
> 状态：designed
> 关联阶段：D0
> 目标门禁：G1
> 当前运行资格：eligible_for_d0

## 1. 目标

完成 D0 阶段最后一项设计契约任务：在 D0-T01 DuckDB 架构边界和 D0-T02 source
registry 初版基础上，定义 D1、D2、D3 三层数据产品的机器可校验契约。

本任务只回答后续数据产品应该有哪些表、每张表的 grain、primary key、required
fields、source requirements、manifest requirements、quality checks、time semantics
和 blocking conditions。它不进入 DuckDB 实现、DDL、loader 或数据采集。

## 2. 非目标

- 不安装 DuckDB；
- 不创建 DuckDB 文件；
- 不写 DDL；
- 不写 loader；
- 不调用任何外部行情、财务、日历、公司行为或指数 API；
- 不保存任何真实数据响应；
- 不生成任何 D1/D2/D3 实体表；
- 不生成 `d3.daily_market_observations` 实体数据；
- 不计算 PCVT、状态、事件、标签、收益或回测；
- 不改变 G0 样本宇宙、时间边界或 source registry 的来源资格结论。

## 4. 仓库边界

DuckDB 只保存结构化登记、标准化事实、可重建派生层和研究入口表。原始响应、原始
文件、供应商导出和抓取日志必须保存在 `data/raw/` 或 `data/external/`，并通过路径、
SHA-256、snapshot id 和 manifest 引用进入 DuckDB。

本任务不得创建 `.duckdb` 文件，不提供 DDL，不安装 DuckDB，不写 loader，不产生任何
正式数据产物。

## 5. 全局要求

所有正式 D1/D2/D3 表必须显式包含：

- `data_version`
- `universe_id`
- `time_segment_id`
- `source_registry_id`
- `source_snapshot_id`
- `run_id`

所有正式 D1/D2/D3 表必须声明：

- grain；
- primary key；
- required fields；
- nullable policy；
- source requirements；
- source snapshot requirements；
- manifest requirements；
- quality checks；
- as-of semantics；
- revision policy；
- blocking conditions。

R0 必须以 `d3.daily_market_observations` 作为唯一正式日频入口，不得绕过 D3 直接读取
D1/D2 表。

## 6. 核心表

本契约定义 9 张核心表：

```text
d1.security_master
d1.trading_calendar
d1.raw_market_prices
d1.corporate_actions
d1.trading_constraints
d2.adjusted_market_prices
d2.market_price_quality_flags
d2.membership_alignment
d3.daily_market_observations
```

各表的 grain、primary key、required fields、来源限制、manifest 要求、质量检查和阻断
条件以 `configs/d0/data_product_contracts.v1.json` 为机器可校验真源。

## 7. 来源约束

- `CSINDEX_OFFICIAL` 只能作为 membership evidence / membership alignment 来源，不得作为行情、公司行为、交易状态或财务报表来源；
- `A_STOCK_DATA_RECON` 不得作为任何正式表来源，只能作为端点调研来源；
- `HITHINK_FINANCIAL_API` 可作为 D1/D2 表设计时的候选来源，但由于当前条款、as-of 和 revision 规则仍待审，不得授权正式 ingestion；
- `BAOSTOCK` 可作为备用或交叉校验候选来源，但不得作为唯一正式主源；复权因子 as-of 语义明确前，不得作为 D2 调整价格正式来源；
- 任意来源若缺少 raw snapshot、SHA-256、as-of policy、revision policy 或 manifest 引用，不得进入正式表。

## 8. D0 关闭语义

本任务完成后，D0 的架构、来源资格、原始快照审计要求和 D1/D2/D3 数据产品契约才算
固定。D0-T03 不授权正式数据采集。

下一步不是正式数据采集，而是 `D1-T00`：DuckDB 依赖、空 schema、DDL 和契约测试。
正式数据采集至少要等到 DuckDB 空 schema 与 source-specific raw snapshot / manifest /
source terms / as-of / revision 审查机制就位之后。

首次采集也只能是极小样本探针，不得直接进行全市场、全历史或 CSI800 全量拉取。

## 9. 验收标准

- D1/D2/D3 九张核心表的数据产品契约完整；
- 所有表的 grain、primary key、required fields、source requirements、manifest requirements、quality checks 和 blocking conditions 明确；
- D3 被明确为 R0 唯一正式日频入口；
- D0-T02 source registry 的来源限制被契约引用，并能阻断未过审来源；
- 配置、schema、测试和文档索引全部通过质量检查；
- PR 不包含 DuckDB 依赖、DuckDB 文件、数据采集代码、loader、worker、行情数据、运行 manifest 或正式数据产物。
