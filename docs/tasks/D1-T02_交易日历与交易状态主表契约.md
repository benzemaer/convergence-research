# D1-T02 交易日历与交易状态主表契约

## 状态

completed via PR #11

## 目标

本任务实现 `d1.trading_calendar` 与 `d1.trading_constraints` 的契约/边界设计。
目标是基于 `configs/d0/data_product_contracts.v1.json` 中两张 D1 表的契约，
明确字段语义、主键、nullable、受控词表、来源边界、as-of/revision 阻塞条件，
以及未来 loader 必须满足的质量检查。

## 非目标

- 不调用任何外部 API。
- 不新增 loader、API client、ETL worker 或 source-specific ingestion 代码。
- 不导入真实交易日历、停牌、ST、涨跌停、交易状态或行情数据。
- 不生成 DuckDB 实体数据，不创建 committed DuckDB 文件。
- 不生成 raw snapshot、dataset manifest、run manifest 或 artifact manifest。
- 不改变 D0 source registry 的来源资格结论。

## 契约边界

- `d1.trading_calendar` 定义交易所交易日与 session 日历，不定义证券级可交易性。
- `d1.trading_constraints` 定义停牌、ST、涨跌停价格和可交易性事实。
- trading status / tradability / suspension / ST / price limit 是交易约束事实，
  不是价格形态信号，不得直接进入 R0。
- R0 只能通过 `d3.daily_market_observations` 使用 D1 约束与日历引用。
- `HITHINK_FINANCIAL_API` 仍仅为候选来源；`BAOSTOCK` 对 trading constraints
  也仅为候选/备选来源，不授权正式 ingestion。
- `CSINDEX_OFFICIAL` 与 `A_STOCK_DATA_RECON` 不得作为交易日历或交易状态正式来源。

## 阻塞条件

- source terms 未完成正式数据使用审查。
- as-of 与 revision 规则未闭环。
- raw snapshot、SHA-256、source snapshot manifest 或 run manifest 缺失。
- 日历 revision policy 或交易状态 observed_at 语义未知。
- 交易状态、停牌、ST、涨跌停或可交易性质量为 unknown 且无法追溯。
