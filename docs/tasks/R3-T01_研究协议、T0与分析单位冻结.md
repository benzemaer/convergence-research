## 3. T0 anchor decision

R3 采用 `natural_exit_attempt`，来源语义为 R2 的 `provisional_component_exit`，
但不再把后者称为最终 release。它表示：一个已经取得 event 资格的事件中，本次
confirmed run 发生合法 natural exit 的实时观察起点。

```text
selected_anchor: natural_exit_attempt
source_r2_candidate: provisional_component_exit
interpretation: real-time observable exit attempt anchor, not final event-zone exit
finalized_zone: rejected_as_T0_due_to_signal_delay_and_event_level_semantics
retrospective_final_exit: rejected_as_T0_due_to_lookahead
event_level_last_exit_observation_time: audit_and_reconciliation_only,
  never_used_to_enumerate_attempts
```

后续 reentry、same-zone requalification、merge、g 超限或 event finalization 都不能
追溯删除已经发生的 T0。`zone_finalization_time`、`last_exit_observation_time` 和
`last_confirmed_end_date` 只能用于事后审计或 reconciliation。

## 4. 合法 natural exit predicate

未来物化必须先在完整 expected-row surface 上按
`state_version_id × security_id × trade_date` 排序计算直接相邻的 `LAG`，然后再做
候选过滤。禁止先删除 invalid、ineligible、missing 或 unknown rows，再寻找上一条
有效行。

设直接前一 expected row 为 `p`、当前为 `c`。合法 T0 必须同时满足：

```text
p.eligible_state=true
p.quality_state=valid
p.confirmed_state=true
p.active_event_id_as_of IS NOT NULL
p.event_status_as_of IN (QUALIFIED_ACTIVE, REENTRY_PENDING_QUALIFICATION)
c expected row 实际存在
c.eligible_state=true
c.quality_state=valid
c.raw_state=false
c.confirmed_state=false
p 与 c 是完整 expected-row surface 上的直接相邻行
```

之后以 `event_id=p.active_event_id_as_of` 连接 event zone，确认 state/security 相同、
`first_qualification_time <= membership_available_time`，并确认当前 membership 行
存在且 availability time 非空。当前 event ID 若非空但不同于 prior ID，必须报
`EVENT_ID_CONFLICT` 等价的 contract failure，不能任选一个 ID。

```text
exit_attempt_date = c.trade_date
exit_attempt_time = c membership.membership_available_time
time_resolution = daily_state_with_point_in_time_availability
```

`natural_state_exit` 必须通过 R2 transition registry 重放得到：

```text
CONFIRMED_ACTIVE → CONFIRMED_EXITED
trigger=raw_false, hard_break=false, reason_code=natural_state_exit
```

事件未取得资格、普通 prequalification raw=false、prior 未 confirmed、prior
invalid/ineligible/missing、current missing/ineligible、quality interruption、raw
unknown、unknown、blocked、diagnostic_required、expected_empty、right/sample-end
censoring、仅由 finalization 推断历史 exit、跨 invalid row 跳跃和跨 event/state
version 继承均不得生成正常 T0。invalid row 后重新出现 valid raw=false 也不得补造。

## 5. 主分析单位与身份

正式主分析单位是：

```text
state_version_id × event_id × exit_attempt_id
all_legal_exit_attempts_are_primary=true
first_attempt_only=false
later_attempts_are_sidecar=false
```

同一 event 内重新 confirmed 并再次 natural exit 时生成新的 attempt；重新达到 d
qualification 和尚未达到 d 的 unqualified reentry 都保留。后者不增加 qualified
component count，但必须保留 `unqualified_reentry=true`。

R2 public tables 没有原始 confirmed-run ID，T01 使用 namespace
`r3_confirmed_run_v1`。同一 state/version/event/security 内连续的
`confirmed_state=true AND component_member=true` expected trading rows 构成一个
source run。其 ID 是对以下 canonical JSON 做完整 SHA-256：

```text
namespace, contract_version, state_version_id, event_id, security_id,
source_component_start_date
```

`source_component_ordinal` 按 `source_component_start_date, source_component_id` 排序，
从 1 开始。`component_count_as_of_exit` 是截至 T0 已知且 qualification availability
不晚于 T0 的 distinct qualified source run 数；因此 source ordinal 可以合法大于该
count。`zone_revision_as_of_exit` 取 T0 membership row 的 zone revision。

`exit_attempt_id` 使用 namespace `r3_exit_attempt_v1`，canonical 字段为
`contract_version、state_version_id、event_id、security_id、source_component_id、
exit_attempt_date、exit_attempt_time`。不得加入 finalization、final count、final
revision、resolution、future path/label 或 boundary version。attempt ordinal 在
`state_version_id × event_id` 内按 `exit_attempt_time、exit_attempt_date、ID` 排序。
重复 ID、重复 source-component/T0、ordinal 不连续、时间排序不一致或同一 T0 被多个
event 认领时必须 fail closed。

## 6. 字段、landmark 与 horizon

field-semantics registry 对每个字段登记 type、nullable、availability class、available
time source、T0/T1/T2 allowed、audit-only、forbidden-model-feature、source artifact
和 derivation rule。state/event/security/attempt/component identity、ordinal、
`component_count_as_of_exit`、T0 date/time、prior/current state、exit reason、g 与
event status 是 T0 可知字段。

`final_component_count`、最终 attempt 数、`event_balanced_weight`、resolution、
same-zone requalification、g-exceeded、future overlap final result、future path class
和 future boundary hit 都是 post-event/post-path 字段，不得作为 T0/T1/T2 特征。

```text
T0 = exit_attempt_date / exit_attempt_time
T1 = T0 后第 1 个有效、可观测 expected trading row
T2 = T0 后第 2 个有效、可观测 expected trading row
```

有效行要求 expected row 存在、`eligible_state=true`、`quality_state=valid`。异常行
不计入 T1/T2 ordinal，但必须记录 intervening count、reason set 和 unavailable
reason，不得静默跳过。Horizon `H ∈ {5,10,20,30}` 从 T0 后第一个有效未来交易日
开始计数，T0 不计入：

```text
H5=early reaction
H10=short-term path formation
H20=primary path-classification horizon
H30=extension and late-reversal check
primary_horizon=H20
```

T01 不生成 PCVT、价格、成交量、环境、return、MFE、MAE、boundary hit、path class，
也不决定 R4 deferred prediction rule；不得用 calendar-day 替代有效交易日。

## 7. sample split 与相关性治理

```text
sample_split_unit=event_id
all_attempts_of_same_event_in_same_split=true
attempt_level_random_split=forbidden
split_roles=design | validation | out_of_sample | purged
purge_embargo_length=32 valid trading days
purge_basis=H30 plus maximum T2 landmark delay
split_topology_status=frozen
calendar_boundary_status=unselected_by_design
calendar_boundary_selection_owner=R3-T02
cluster_key_primary=event_id
cluster_key_robustness=security_id
future_window_overlap_marker_required=true
attempt_weight=1.0
event_balanced_weight=1 / final attempt count within event
```

同一 event 的所有 attempts 必须进入同一 split。允许用于日期边界选择的信息只有
date/event/attempt/observability coverage；future labels、returns、boundary hits、
model accuracy 和 trading results 禁止。R3-T02 必须在 control matching、standardization
或 empirical-boundary estimation 之前登记并冻结具体日期边界。

语义时，必须创建新的 R3-T01 contract version，并重新审阅。改变 R2 state/version、
confirmed interval、d、g 或 event identity 时，最低退回 R2。R3-T02 只有在本
Implementation 被用户审阅接受并完成后续授权后才可开始。
