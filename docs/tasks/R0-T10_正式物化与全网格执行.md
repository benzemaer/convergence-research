# R0-T10 正式物化与全网格执行

状态：phase 1 pre-full-grid gate in progress；正式 27 组 full-grid 等待本提交审核后另行执行。

## 目标

R0-T10 补齐 R0-T09 之后仍缺失的正式执行链路：定位真实授权研究输入，生成或接收 R0-T04 至 R0-T07 的正式 upstream result artifacts，生成 R0-T09 `authorized_input_manifest.json`，并在通过 dry-run 与 baseline 单组 materialization 后执行 27 组 full-grid production materialization。本 PR 按提交策略先完成 full-grid 前的代码、契约、门禁、测试和执行记录框架，明确阻止在审核前启动正式 27 组 full-grid。

## 非目标

R0-T10 不生成 R0 审计报告，不生成 R1 handoff，不计算未来收益标签、未来波动、突破方向、释放方向、backtest、portfolio、trade signal、gap merge 或 cooldown。R0-T10 也不改变 R0-T04 至 R0-T09 的指标、分位、状态、确认、主网格配置、V1 baseline 或 worker 上限定义。

## 输入

正式输入只能来自真实授权的 R0-T04 至 R0-T07 upstream artifacts。禁止把 `tests/fixtures/*`、unit test helper、synthetic fixture、`synthetic_smoke_fixture`、`_contract_grid_payload()`、tmpdir test outputs 或手工伪造 JSON 作为 production formal input。若本地只存在 D2/D3 candidate 或 handoff 报告，且报告仍显示 `formal_use_authorized=false`、`pcvt_values_generated=false` 或 `r0_state_generated=false`，R0-T10 必须返回 blocked，不得追认为正式 R0 upstream。

R0-T10 入口为：

```bash
python scripts/r0/run_r0_t10_formal_materialization.py \
  --output-dir data/generated/r0/r0_t10/<run_id> \
  --run-id <run_id> \
  --code-commit <git_sha> \
  --r0-t04-input <r0_t04_raw_metric_results.json> \
  --r0-t05-input <r0_t05_score_results.json> \
  --r0-t06-input <r0_t06_nested_daily_state_results.json> \
  --r0-t07-input <r0_t07_confirmation_results.json> \
  --dry-run-r0-t09 \
  --baseline-r0-t09 \
  --max-workers 2
```

在本 phase 1 提交中，`--full-grid-r0-t09` 会固定返回 blocked，reason 为 `full_grid_requires_second_submission`。

## 输出

R0-T10 的正式输出目录为 `data/generated/r0/r0_t10/<run_id>/`。full-grid 前阶段最多写出 upstream generation summary、R0-T09 input manifest 及 dry-run/baseline 输出；正式 27 组 full-grid 输出只能在第二次提交审核通过后生成。本 PR 不提交任何 `data/generated/r0/r0_t10/**/*.json`、DuckDB、CSV gzip、DONE/FAILED、manifest、logs 或 partial 文件。

预期路径为：

```text
data/generated/r0/r0_t10/<run_id>/upstream/
  r0_t04_raw_metric_results.json
  r0_t05_score_results.json
  r0_t06_nested_daily_state_results.json
  r0_t07_confirmation_results.json
  upstream_generation_summary.json
data/generated/r0/r0_t10/<run_id>/r0_t09_inputs/
  r0_t09_full_grid_payload.json
  authorized_input_manifest.json
  generation_summary.json
data/generated/r0/r0_t10/<run_id>/r0_t09_full_grid/
  ...
```

## 契约与门禁语义

契约固化在 `configs/r0/r0_t10_formal_materialization_contract.v1.json`，schema 固化在 `schemas/r0/r0_t10_formal_materialization_contract.schema.json`。`real_data_read_authorized=true`、`generated_json_write_authorized=true`、`r0_t09_runner_execution_authorized=true`，但 phase 1 明确 `r0_t09_full_grid_execution_authorized_in_phase_1=false`。`max_workers_default=2`，`max_workers_upper_bound=2`，超过 2 必须拒绝。

正式 upstream 必须包含 `raw_metric_results`、`indicator_score_results`、`dimension_score_results`、`nested_daily_state_results`、`daily_confirmation_results` 和 `confirmed_interval_results`。R0-T09 builder 继续负责 coverage guard，要求 9 个 `(W,q)`、108 个 `(W,q,K,state_name)`，K 仅为 2/3/5，V1 仍为 `V1_TurnoverShrink20_60` / `TurnoverShrink20_60_raw`。

## 验收标准

Phase 1 验收标准是：缺失正式上游时必须 blocked 且不写 authorized manifest；fixture/synthetic 路径必须被拒；显式合法 upstream 输入可生成 R0-T09 manifest 并通过 dry-run 与 baseline 单组 materialization；`--full-grid-r0-t09` 在本提交中必须 blocked；所有输出不得包含 audit report、R1 handoff、future、return、backtest、portfolio 或 signal 字段。

## 失败状态

缺少任一 upstream 输入、输入 JSON 无效、row set 缺失或为空、路径疑似 fixture/synthetic、D2/D3 报告显示 formal use 未授权、PCVT 值未生成或 R0 state 未生成时，R0-T10 返回 `status=blocked`，写出 summary，但不得生成 formal manifest 或 full-grid artifact。baseline 单组失败时，R0-T10 保留 R0-T09 FAILED/log 供排查，不得继续 full-grid。
