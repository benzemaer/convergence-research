# D3-T11 量额股本换手字段全量候选物化与数据更新

## 状态

completed via PR #59；D3-T12 对 D3-T07 auto fallback 的 warning 接受语义作后续修正。

## 目标

D3-T11 基于 D3-T10 字段契约，生成量额、股本、换手率和相关质量状态的 D3 candidate 数据层。该 task 读取 D3-T07 candidate daily observation，按证券主轴调用 `tnskhdata` `daily_basic`，复用 D3-T07 的 `vol` / `amount` 作为 daily 输入，并严格按 D3-T10 单位规则标准化为 `volume_shares`、`amount_yuan`、`daily_vwap`、`total_share_shares`、`float_share_shares`、`free_share_shares`、`turnover_float` 和 `turnover_free`。

本 task 输出 candidate DuckDB、字段覆盖率、质量报告、provider call summary 和兼容 handoff report，供后续消费 task 审计。它不发布 formal data_version，也不计算 PCVT raw metrics。D3-T11 只生成跨研究开放 candidate observation layer 的补充字段，不判断该 candidate 是否已满足 R0-R6 任一阶段的消费条件。

## 非目标

本 PR 不生成 PCVT percentiles、scores、states 或 intervals，不生成 future labels、future returns、breakout direction、backtest 或 portfolio，不修改 R0-T01 weak baseline，不直接替换 `V1_VolShrink20_60` / `V2_AmountLevel20Pct`，不提交 provider raw payload、token、cookie、credential、DuckDB、CSV、JSON generated data artifact 或 formal manifest。

## 输入

必需输入包括证券列表、日期范围、D3-T07 candidate DuckDB 和 `tnskhdata` token 环境变量。token 只能从 `TNSKHDATA_TOKEN`、`TUSHARE_TOKEN` 或 `TNS_TOKEN` 读取，不得硬编码。D3-T07 输入必须是 `data/generated/d3/d3_t07_candidate_daily_observation/d3_t07_candidate_daily_observation.duckdb` 边界内的 candidate artifact，不得读取 `data/raw`、`data/external`、MarketDB、`.day`、D1 或 D2 generated outputs。

D3-T11 默认读取 D3-T07 candidate observation，即 `data/generated/d3/d3_t07_candidate_daily_observation/d3_t07_candidate_daily_observation.duckdb`。若该 D3-T07 candidate DuckDB 不存在，runner 可在 provider 调用前检查 D2-T20 candidate 的默认三件套，并调用 `generate_d3_t07_candidate_daily_observation()` 自动生成 D3-T07，再继续执行 D3-T11。D3-T11 必须接受 D3-T07 的 `accepted_candidate_observation` 和 `accepted_candidate_observation_with_warnings`；若 D3-T07 仅因 `policy_evidence_pending_hash` 等 soft warning 处于 with-warnings 状态，D3-T11 继续进入 `daily_basic` fetch，并在报告中记录 evidence pending、`formal_use_authorized=false` 和 `consumer_readiness.evaluated_by_d3=false`。D3-T11 不得直接复制 D2-T20 DuckDB 为 D3-T11 输出，也不得直接在 D2-T20 DuckDB 上写 D3-T11 表。

默认运行入口使用 `configs/d2/csi800_static_2026_06_membership_alignment.v1.json` 作为股票池，调用 `resolve_security_provider_codes()` 将 `rows[*].security_id` 解析为 `tnskhdata_ts_code`。如果解析后证券数低于 790，默认阻塞远程调用；只有显式传入 `--allow-low-security-count` 才可继续。runner 也支持 `--securities-file` 作为显式样本或 operator-provided 股票列表，并支持 `--env-file` 加载 `.env.local` 中的 token，加载过程不得打印或写入 token 值。

## 输出

允许本地输出到 ignored 目录 `data/generated/d3/d3_t11_volume_amount_share_turnover_candidate/`。建议输出包括 `d3_t11_volume_amount_share_turnover_candidate.duckdb`、`d3_t11_generation_summary.json`、`d3_t11_field_coverage_report.json`、`d3_t11_quality_report.json`、`d3_t11_provider_call_summary.json`、missing/crosscheck/vwap CSV、`d3_t11_r0_handoff_report.json` 和 `d3_t11_file_hash_summary.json`。这些 generated outputs 不提交到 Git。

## 字段与标准化规则

`volume_raw = daily.vol`，单位为手，`volume_shares = volume_raw * 100`。`amount_raw = daily.amount`，单位为千元，`amount_yuan = amount_raw * 1000`。当 `volume_shares > 0` 时，`daily_vwap = amount_yuan / volume_shares`。

`daily_basic.total_share`、`float_share`、`free_share` 的单位均为万股，分别标准化为 `total_share_shares`、`float_share_shares`、`free_share_shares`。`turnover_float = volume_shares / float_share_shares`，`turnover_free = volume_shares / free_share_shares`，并用 provider `turnover_rate` 与 `turnover_rate_f` 做交叉校验。

## 公司行为与可比性政策

D3-T11 不计算 PCVT，但必须保留后续 R0 需要的保守可比性字段：`corporate_action_types_in_window`、`share_comparability_corporate_action_in_window`、`adjusted_vwap_policy`、`common_corporate_action_basis_policy`、`common_share_basis_policy` 和 `volume_comparability_policy`。如果当前 candidate 不能完整识别 window-level 公司行为类型，必须输出 `unknown` 或 `policy_pending`，不得把未知解释为可比较。

## Gate 规则

runner 必须支持 token 缺失时 clean exit，并输出 `blocked_missing_tnskhdata_token`，不得 traceback。真实 provider raw payload 不得写入 report 或 Git。输出目录必须位于 `data/generated/d3/`；不得写入 `data/raw`、`data/external`、MarketDB、`.day`、D1/D2 generated outputs 或正式 manifest/data_version 路径。

如果默认 D3-T07 source DuckDB 不存在且 auto fallback 关闭，runner 必须在 provider 调用前返回 `blocked_missing_d3_t07_source`。如果 auto fallback 开启但 D2-T20 默认输入不完整，runner 必须返回 `blocked_missing_d2_t20_inputs`。这些阻塞状态都必须记录 `remote_provider_called=false`、`candidate_generated=false`、`pcvt_values_generated=false`、`r0_state_generated=false` 和 `formal_data_version_published=false`。

D3 candidate generation 不等于 formal release，也不等于任一 R-stage readiness。D3-T11 报告必须拆分 `d3_candidate_generated` / `d3_candidate_generation_decision` 与通用 `consumer_readiness`。R0-R6 或未来研究路线由各自消费 task 定义 consumer readiness profile；D3-T11 不输出 `r1_ready_candidate` 至 `r6_ready_candidate`，也不使用 `r0_ready_candidate` 作为 candidate generation decision。若为兼容旧报告暂时保留 `r0_ready_candidate`，必须标记 deprecated。

## Retry failed provider calls

D3-T11 runner 支持 `--retry-failed-only`，仅用于恢复 transient provider failures。该模式读取既有 D3-T11 generated outputs，包括 candidate DuckDB、generation summary 和 provider call summary；它不会清空 output-dir，也不会重新调用已经成功的 799 个 `daily_basic` 请求。默认从 `d3_t11_provider_call_summary.json` 读取 `status = failed` 且 `endpoint = daily_basic` 的 calls；可用 `--retry-ts-code` 将 retry 限定到一个或多个指定股票，例如 `002153.SZ`。

retry 成功后，runner 会从 D3-T07 candidate DuckDB 重新读取对应 `ts_code` 的 source rows，用新拉取的 `daily_basic` rows 重建该股票全日期 D3-T11 candidate rows，并通过 `INSERT OR REPLACE` 更新既有 D3-T11 candidate DuckDB。随后必须从更新后的完整 D3-T11 DuckDB 重新生成全量 coverage、quality、provider、handoff 和 hash reports；不得只输出单股票局部 reports。

该模式仍不发布 formal data_version，不计算 PCVT raw metrics，不生成 PCVT percentiles/scores/states/intervals、future labels、future returns、breakout direction、backtest、portfolio 或 R state。retry 生成的 DuckDB/CSV/JSON、provider raw payload、token、cookie、credential 和 `.env.local` 仍不得提交。
